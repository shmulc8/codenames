# Missing UI Gaps — Hebrew Codenames Copilot

Notes from reviewing `page-1.jpeg`, `page-2.jpeg`, and `page-3.jpeg` (screenshots of the `shmulc/hebrew-codenames-copilot` Space).

> **Verification note (2026-07-16):** this file was reviewed by a second pass (Opus) that read the Space's actual public source (`app.py`, `methods.html`) instead of only screenshots/rendered HTML. That pass corrected two items that were previously listed as missing features but actually already exist, and flagged several others as unverified inference rather than confirmed fact. Corrections are marked inline below.

## Critical / functional gaps

- ~~No way to mark words as guessed~~ — **CORRECTED.** This exists: the "🃏 חשיפה" (reveal) action flips a card and the backend excludes revealed cards from the engine's board on the next clue (confirmed in `app.py`: revealed cards are passed and excluded from candidate scoring). The real gap is **discoverability** — "חשיפה" doesn't read as "mark this word as claimed" to a new user, and there's no distinct visual state (grayed out / checkmarked) shown in any screenshot to confirm what a revealed tile looks like. Treat this as a labeling/affordance fix, not a missing feature.
- ~~No auto-clustering mode~~ — **CORRECTED.** This exists: the clue-request panel's word-selection field is optional — the page-1 copy itself says "או השאר ריק והעוזר יבחר את הקבוצה הטובה ביותר בעצמו" (or leave it empty and the assistant will choose the best group itself). Leaving the selection empty triggers the engine to scan the whole board and propose its own cluster. The real gap is that this behavior is implicit and easy to miss — there's no visible button or prompt telling the user that skipping selection is a deliberate "auto-pick for me" mode.
- **No secrecy/access control on a shared screen.** The Spymaster/Guesser toggle is just a button — anyone at a shared device can click "אני רב המרגלים" and see the full color key. Needs a blur-until-confirmed step, separate device/link per role, or a PIN. (Confirmed independent of the Change 1 plan below — see `ui-changes.md`.)
- **Board origin is unclear and may contradict the core product idea.** The 🎲 "לוח חדש" button generates a random board (confirmed: the `/api/deal` endpoint just samples words, no image ingestion anywhere in the source). The project's premise is photographing an actual physical board/key card so the assistant matches reality — there is no photo-import path in this app at all. This is the single biggest product-level mismatch found and should stay top priority.
- **No clue history — unverified, likely true but not confirmed.** No server endpoint replays prior clues back to the client; clue/feedback logging appears append-only and isn't surfaced in the UI. Not confirmed against the full frontend (`copilot.html`) though, so treat as "not surfaced," not "doesn't exist anywhere."
- ~~Team-count legend may not reflect live game state~~ — **RESOLVED BY DECISION, not by verification.** Rather than confirm whether the numbers update live, the owner decided to drop the numeric counts entirely and de-emphasize the legend generally — see `ui-changes.md` Change 3.
- **Not mobile compatible.** Fixed 5-column board, dense side panels, and small icon buttons show no real responsive/breakpoint behavior. Confirmed on the live site: it shows a dismissible top banner reading "החוויה המלאה (מפת המשמעויות, לוח רחב) הכי טובה במסך מחשב — אבל אפשר גם כאן" (the full experience is best on desktop, but you can also use it here) — i.e. the team already knows mobile is degraded and ships a disclaimer instead of a fix. The semantic map and wide board are explicitly called out as compromised on small screens. See `ui-changes.md` Change 6 for the proposed fix — a pan-and-zoom board canvas that keeps cards full-size on phones (the user moves around the board instead of every tile being shrunk), rather than shipping a disclaimer.

## Clarity / labeling problems

- **Unlabeled confidence numbers — and one mislabel corrected.** The "100 / 83 / 67…" bars in the guesser ranked list are word-similarity scores with no unit or explanation — that part stands. But "10 / 1" and "10 / 2" (next to "אפשרויות אחרת" / other options, page-1 and page-3) are **not** a confidence score — they're a shortlist position, "option N of 10" (the engine caps its alternate-clue list at 10). The two numbers were conflated in an earlier pass of this doc; they need two different fixes: label the 100/83/67 bars with a score legend, and label "10/1" as an option counter (e.g. "אפשרות 1 מתוך 10").
- **Unlabeled small teal underline bars** under every board tile (visible in page-2/page-3) — likely encode something (confidence? heat?) but there's no legend anywhere.
- **Mixed language register.** Technical labels ("fastText," "LLM ללא," "App / Files / Community") sit inside an otherwise fully Hebrew, consumer-facing UI — inconsistent for a casual game-night tool.
- **Numbered badges on cards** (page-2) are explained only by small caption text ("המספרים מסמנים את סדר הניחוש המומלץ") placed far from the badges themselves — easy to miss.
- **No intuitive "this card is already out of play" state.** The reveal action ("🃏 חשיפה") does mark a card as guessed and the backend already excludes revealed cards from future scoring (see the corrected item at the top of this doc) — but there is no clear, at-a-glance visual language on the board for *which cards are already chosen and therefore no longer in play*. Across page-1/2/3 there is no confirmed grayed-out / struck-through / dimmed tile state, so a player can't glance at the board and instantly see the shrinking set of live words. This matters more than a normal labeling nit because the whole point of the tool is reasoning over the *remaining* words: if the human can't tell at a glance which cards the engine has already dropped, they lose trust that the hint accounts for the real current board, and they can't cross-check the engine's view against the physical game. The current affordance (a small toggle action + implicit backend exclusion) is not intuitive enough — "already chosen" needs to be a first-class, obvious visual state, distinct from team color, distinct from selection, and ideally showing *which team* claimed each spent card. Reconcile with the reveal-discoverability fix (top of doc) and the per-role visibility checkboxes (Layout section) — all three touch the same board state and should be designed as one system, not three. See `ui-changes.md` Change 4 for the implementation spec.

## Accessibility

- **Color-only team encoding — confirmed, act on regardless of further verification.** Teal (mine) / red (opponent) / black (assassin) / gray (neutral) rely on hue alone with no icon, pattern, or shape backup. This is a real WCAG 1.4.1 (use-of-color) failure, not just a stylistic nit — the assassin tile is the highest-stakes one to misread. Note it shows up in two different render styles across the screenshots (filled tiles on page-1, colored outlines on page-3) — any fix needs to cover both.
- **RTL badge placement — polish, unverified.** Numbered badges sit at the top-left of cards, which is the "last" corner an RTL reader's eye reaches — top-right would follow reading direction more naturally. This is a reasonable heuristic, not a confirmed accessibility failure (never checked with a screen reader) — keep it as a minor polish item, not on par with the color-only issue above.

## Layout / information architecture

- **Toolbar overload.** Risk toggle (מאוזן/זהיר/נועז), reveal, edit pencil, clear-selection X, new board, and more are crammed into one row with no grouping — the risk toggle is functionally tied to clue generation but spatially disconnected from the clue box.
- **Semantic-space graph is underused — but it's core, don't shrink it.** As rendered ("המרחב הסמנטי" / the semantic space, subtitle "הרמז יושב במרכז המילים שלך" — the hint sits at the center of your words), it spends large screen real estate on 2-3 labeled dots plus scattered unlabeled colored dots. An earlier draft of this item suggested *either* make it interactive *or* shrink it — correcting that: per `CLAUDE.md` this map ("showing me a semantic grid... which other words have high odds of being chosen and their chance") is a headline feature of the product, not filler to shrink. The real gap is that it's under-built: most dots are unlabeled and inert, there's no similarity readout, no way to tell why a dot is where it is, and no link between a dot and its board tile. Fix by making it fully interactive and legible, not smaller. See `ui-changes.md` Change 7 for the spec.
- **Ambiguous button scope.** "✏️ ערוך" and "🃏 חשיפה" do carry text labels (correcting an earlier note that assumed icon-only), but the labels don't explain scope — edit the word list, the team assignment, or both? "חשיפה" (reveal) is now confirmed to be the mark-as-guessed action (see corrected item above) — its label should say so more directly, e.g. something closer to "סמן כנחשף" (mark as revealed).
- **Weak visual differentiation between Spymaster and Guesser views.** Near-identical button styling for very different actions ("clue for words I picked" vs. "what should I guess?") risks mis-clicks when flipping roles under time pressure.
- **The desktop board looks nothing like the real game — a missed fidelity/delight opportunity.** The web board is a flat, abstract grid of colored web tiles; the physical game (see `real-board-reference.jpeg` — a 5×5 tray of cream word-cards and illustrated agent tiles) is tactile and instantly recognizable. On desktop, where there's ample screen room, the app currently spends that room on an abstract layout instead of evoking the tabletop it's meant to mirror. This matters beyond aesthetics: the product premise is that the on-screen board *is* the player's physical board, so making it visually resemble the real thing (recessed tray, card-stock word tiles, illustrated agent portraits for revealed cards) reinforces the screen↔table mapping the whole tool depends on. The illustrated agent tiles in the photo are themselves the physical analog of Change 4's "chosen/out-of-play" state — art tint (blue team / red team / beige bystander) encodes who claimed each spent card. See `ui-changes.md` Change 5 for the desktop "real board" skin spec (and a toggle back to the compact view for users who prefer it).
- **Missed element, now documented: per-role color-layer checkboxes.** Bottom-right of the board there are separate checkboxes for "הצוות שלי / היריב / ניטרלי / מתנקש" (my team / opponent / neutral / assassin). This is a partial reveal/visibility control that neither this doc nor `ui-changes.md` accounted for in the original pass. It's directly relevant to both the shared-screen secrecy gap above and the "mark as guessed" discoverability fix — reconcile all three before designing new secrecy or reveal UI, since this checkbox row may already be doing part of that job.

## Additional findings from live-site testing (huggingface.co/spaces/shmulc/hebrew-codenames-copilot)

- **Risk dial has zero inline explanation — but keep the fix plain-language, not technical.** "זהיר / מאוזן / נועז" (careful/balanced/bold) changes engine behavior with no tooltip or microcopy in the app itself. On review, the earlier draft of this item asked for the actual engineering parameters (coverage size, safety margin, penalty weights) to be surfaced inline — that's over-indexing on the `/methods` deep-dive page. The right fix is a one-line, plain-language version of what the methods page itself already summarizes in words: cautious = safety-first, balanced = safe coverage, bold = maximum reach. No numbers needed in-app.
- **No visible empty/failure state — confirmed, keep as-is.** The engine can and does flatly return a "no safe clue" result (`no_clue=True` with a Hebrew note, confirmed in `app.py`) at cautious risk. Neither screenshot shows this state rendered — worth designing explicitly. This is the one item in this section confirmed against source rather than the methods page alone.
- **Known model limitations — lower priority, wrong venue.** The methods page is candid about failure modes (polysemy, loanwords losing their vector, lexicon gaps), but surfacing this mid-game is noise, not help — a player trying to give a clue doesn't need an ML caveat in the moment. This belongs in the existing "איך זה עובד" link, which already covers it; not worth new in-app UI.
- **No vocabulary-coverage warning.** The clue engine draws candidates from a fixed ~5,260-word list. Lower priority than originally listed, since it's downstream of the bigger "no photo-import" gap above — until boards can come from a real photo, this scenario (an out-of-vocabulary word from *your* physical board) can't actually occur in this app yet.

## Open questions raised during review

- How do you update the board as words get chosen during the actual game?
- Can the tool generate a hint for a word set *it* chooses, not just one the user manually selects?
- Is this meant for a shared single device, or synced across multiple players' devices?
