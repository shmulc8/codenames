# UI Fix Memory

## UIFIX-001 — Restoring a revealed tile leaves stale outcome data

- **Severity:** High
- **Manual reproduction:** On the demo board, select tiles 0 and 1, request a clue, mark it in use, click `btn-lifecycle-0` to reveal tile 0, then click it again to restore the tile before the five-second outcome debounce expires.
- **Observed evidence:** The tile returned to `inPlay`, but `clue.used.revealedAfter` and the matching session-log entry still contained tile 0. After the debounce, `window.__lastFeedback.revealed` incorrectly contained tile 0.
- **Root cause:** `toggleLifecycle` appended a reveal on the `inPlay` → `chosen` transition, but the `chosen` → `inPlay` transition never removed the word from the active `UsedClue` or its corresponding log entry.
- **Files changed:** `ui/src/state/store.ts`, `ui/tests/e2e/feedback.spec.ts`, `.ui-fix-memory.md`.
- **Fix:** Restoring a tile now filters that word from the active clue's `revealedAfter` array and synchronizes the matching log entry. Revealing adds the word only when it is not already recorded, so a restore/re-reveal cycle contains exactly one reveal.
- **Verification:** Manual Playwright replay on port 5155 confirmed the restored tile was `inPlay`, both active-clue and log `revealedAfter` arrays were empty, and no feedback payload existed after 5.3 seconds. `PW_PORT=5155 npm run typecheck` passed. After `git pull --no-rebase origin feature/ui` reported the branch up to date with no conflicts, `PW_PORT=5155 npm run test:e2e` passed all 50 tests, including the new regression.

## UIFIX-002 — Late OCR completion erases manual board corrections

- **Severity:** High
- **Manual reproduction:** Upload a 1px PNG through `photo-input-board`, wait for `מזהה את מילות הלוח…`, immediately click `דלגו על צילום הלוח — הקלידו ידנית`, and fill `ocr-cell-0` with `בדיקה` while OCR is still running.
- **Observed evidence:** About five seconds later, the old OCR attempt succeeded and replaced `בדיקה` with an empty value.
- **Root cause:** `handleBoardFile` unconditionally applied asynchronous OCR success and failure state. The manual-skip action, a newer upload, and unmounting the setup screen did not invalidate the older in-flight attempt, so stale results could replace words/confidences/status or display a stale error toast.
- **Files changed:** `ui/src/features/photo/index.tsx`, `ui/tests/e2e/photo.spec.ts`, `.ui-fix-memory.md`.
- **Fix:** Board OCR attempts now receive a monotonically increasing request ID. Manual skip, a newer upload, and component unmount invalidate older IDs; only the current attempt may apply recognized words, confidences, status, or error toasts. OCR worker warmup and the active attempt remain unchanged.
- **Verification:** A deterministic Playwright regression on port 5155 blocked the Hebrew trained-data request, skipped to manual input, entered `בדיקה`, released the OCR worker, observed a completed late `recognize` response, and confirmed the manual value and ready state remained intact with no toast. A separate manual Playwright replay with the real worker still showed `בדיקה`, status `מנוע הזיהוי מוכן`, and zero toasts after five seconds. After `git pull --no-rebase origin feature/ui` reported the branch up to date with no conflicts, `PW_PORT=5155 npm run typecheck` passed and `PW_PORT=5155 npm run test:e2e` passed all 51 tests. `git diff --check` passed.

## UIFIX-003 — Failed replacement check leaves stale safety guidance visible

- **Severity:** High
- **Manual reproduction:** On the demo board, open the Check tab and successfully check `טבע`. Override the next `/api/coach/check` response with HTTP 500 and `{ error: 'בדיקה זמנית נכשלה' }`, change the input to `חדש`, and submit it.
- **Observed evidence:** After the error toast, `check-input` contained `חדש`, but `check-result` still showed the complete prior `טבע` analysis, `window.__store.getState().checkedClue` remained `טבע`, and the semantic map retained `aria-label="מפה סמנטית עבור הרמז טבע"`. The UI therefore presented old safety guidance as if it applied to the new clue.
- **Root cause:** A valid check submission retained the prior local `result` and `submittedClue` plus the shared `checkedClue` while awaiting the request, and only replaced them after success. The failure path displayed a toast without invalidating those stale values.
- **Files changed:** `ui/src/features/check/index.tsx`, `ui/tests/e2e/check.spec.ts`, `.ui-fix-memory.md`.
- **Fix:** Once submission validation succeeds, the check panel now clears the previous result, submitted clue, shared checked clue, and stale hover before starting the replacement request. A failure consequently leaves only the error toast, while the successful response path remains unchanged.
- **Verification:** `PW_PORT=5155 npm run typecheck` passed. `PW_PORT=5155 npx playwright test tests/e2e/check.spec.ts` passed all 8 Check tests, including the success-then-failure regression. After `git pull --no-rebase origin feature/ui` reported the branch up to date with no conflicts, `PW_PORT=5155 npm run test:e2e` passed all 52 tests. A separate manual Playwright replay at 1320×900 on port 5155 ended with input `חדש`, `checkedClue: null`, zero `check-result` and `map-hint-node` elements, the neutral board-map label, and only the `בדיקה זמנית נכשלה` toast. `git diff --check` passed.

## UIFIX-004 — Late key-card classification overwrites manual key corrections

- **Manual reproduction:** Delay the browser image-load callback by about 1.2 seconds, upload a solid-red SVG through `photo-input-key`, immediately click `דלגו על צילום המפתח — סמנו ידנית`, and click `key-cell-0` once to assign the assassin role before the old classifier settles.
- **Observed evidence:** When the delayed classification completed, `key-cell-0` changed from assassin to red, all 25 roles became red, and the stale success toast appeared even though the user had explicitly switched to manual assignment.
- **Root cause:** `handleKeyFile` applied every asynchronous success, failure, and `finally` result unconditionally. A newer upload, manual key-photo skip, and component unmount did not invalidate an older in-flight classifier, so stale work could replace roles, change the busy state, and display a misleading toast.
- **Files changed:** `ui/src/features/photo/index.tsx`, `ui/tests/e2e/photo.spec.ts`, `.ui-fix-memory.md`.
- **Fix:** Key-card classifications now receive a monotonically increasing attempt ID. A newer upload, manual skip, and component unmount invalidate earlier IDs; only the current attempt may apply classified or fallback roles, show success/error toasts, or clear the busy state. Manual skip immediately clears the active busy state while preserving the selected preview and manual role workflow.
- **Verification:** A deterministic Playwright regression gates the real `HTMLImageElement` load callback, switches to manual assignment, releases the old classifier, and observes one delayed image release plus all 25 real canvas samples while asserting the manual assassin role survives, the remaining roles stay neutral, no stale toast appears, the busy label stays cleared, and the preview remains. `PW_PORT=5155 npm run typecheck` passed and `PW_PORT=5155 npx playwright test tests/e2e/photo.spec.ts` passed all 10 PhotoSetup tests. On that merged tree, typechecking passed and `PW_PORT=5155 npm run test:e2e` passed all 54 tests. A separate manual Playwright replay at 1320×900 on port 5155 finished with the corrected first cell still assassin, the other 24 cells neutral, zero toast, loading cleared, and the preview retained. `git diff --check` passed.

## UIFIX-006 — Fast clue mock makes loading-state coverage nondeterministic

- **Severity:** Medium (test reliability / blocking CI)
- **Observed evidence:** `PW_PORT=5155 npx playwright test tests/e2e/clue.spec.ts:24 --repeat-each=3` failed two of three runs because the fast mock response completed between the sequential loading assertions. By the time Playwright checked `btn-auto-cluster` or `risk-balanced`, the application had correctly re-enabled the control.
- **Manual evidence:** A separate Playwright run wrapped the real `/api/coach/spymaster` fetch response in an explicit promise gate. While delivery of the resolved response to application code was held, both `btn-auto-cluster` and `risk-balanced` reported `isDisabled: true`, and `risk-balanced.matches(':disabled')` was `true`; releasing the gate produced the expected successful clue result.
- **Root cause:** The test attempted to observe a transient loading state against an intentionally fast mock response without controlling request completion. Its sequential assertions could race with a correct request lifecycle, producing false failures.
- **Files changed:** `ui/tests/e2e/clue.spec.ts`, `.ui-fix-memory.md`.
- **Fix:** The first clue test now holds delivery of the resolved real mock response to application code, waits until that response is held, asserts the spinner and both disabled controls while the application request is still pending, and releases the gate in a `finally` block before validating the existing successful result. Production behavior and the disabled-state assertions are unchanged.
- **Verification:** `PW_PORT=5155 npx playwright test tests/e2e/clue.spec.ts -g "auto-cluster renders option 0" --repeat-each=5` passed all 5 stress runs. `PW_PORT=5155 npx playwright test tests/e2e/clue.spec.ts` passed all 7 Clue tests and `PW_PORT=5155 npm run typecheck` passed. `git pull --no-rebase origin feature/ui` reported the branch up to date with no conflicts, then `PW_PORT=5155 npm run test:e2e` passed all 55 tests. ## UIFIX-007 — An old in-flight clue response is presented as current after the board changes

- **Severity:** High (clue safety)
- **Manual reproduction:** Load and confirm the demo board, hold delivery of the real resolved `/api/coach/spymaster` response, and start auto-cluster. Confirm the held request contains tile 0 (`אריה`), then click `btn-lifecycle-0` while the request is pending and release the response.
- **Observed evidence:** The tile remained `chosen`, but applying the old response changed `window.__store.getState().clue.stale` from `true` back to `false`, rendered the clue `טבע` without the stale overlay, and left the use-clue action available even though the engine had evaluated a live board that still contained `אריה`.
- **Root cause:** `requestClue` did not retain or validate the live-board snapshot used for the request. Although the lifecycle mutation marked clue state stale, the later `setClueResult` call unconditionally reset `stale` to `false` as it installed the response.
- **Files changed:** `ui/src/features/clue/index.tsx`, `ui/src/state/store.ts`, `ui/tests/e2e/clue.spec.ts`, `.ui-fix-memory.md`.
- **Fix:** Each clue request now retains the exact live board it sends and compares its ordered words plus each word's absolute role against the current live board when the response resolves. `setClueResult` accepts a type-safe optional stale value (defaulting to the existing `false` behavior), so a result for a changed board is retained behind the existing stale overlay and cannot be used. Recalculation still builds a fresh request from the current board and clears the warning only after that matching response arrives.
- **Verification:** The new deterministic Playwright regression failed against the original behavior because the stale overlay was absent. With the fix, it proves the held request contains `אריה`, changes that tile to `chosen`, releases the old response, observes `clue.stale === true`, the stale overlay, and a disabled use action, then recalculates and proves the fresh request excludes `אריה` before the result becomes usable. `PW_PORT=5155 npm run typecheck` passed and `PW_PORT=5155 npx playwright test tests/e2e/clue.spec.ts tests/e2e/feedback.spec.ts` passed all 15 focused tests. `git pull --no-rebase origin feature/ui` reported the branch up to date with no conflicts, then `PW_PORT=5155 npm run test:e2e` passed all 56 tests. A separate manual Playwright replay at 1320×900 on port 5155 observed the held old request containing `אריה`, stale state plus the warning and disabled use after the lifecycle change, then a recalculated request excluding `אריה`, cleared stale state, and an enabled use action. `git diff --check` passed; package and lock files were unchanged.

## UIFIX-008 — An in-flight safety check is applied after its board snapshot becomes obsolete

- **Severity:** High (safety-check consistency)
- **Manual reproduction:** Load and confirm the demo board, open Check, hold delivery of the real resolved `/api/coach/check` response, submit `טבע`, and confirm the held request contains tile 0 (`אריה`). While the response is held, click `btn-lifecycle-0` to mark `אריה` chosen, then release the response.
- **Observed evidence:** The old response was accepted after the live board changed: `checkedClue` became `טבע`, `check-result` and `ranked-row-אריה` rendered, and the semantic map was labeled for `טבע`, with no warning or retry toast. The visible safety analysis therefore included a card that was no longer in play.
- **Root cause:** The check submission built its request from `liveBoard(state)` but did not retain or validate that snapshot before applying the asynchronous response. Success always updated the local result, result cache, shared checked clue, and semantic-map hint regardless of intervening lifecycle changes.
- **Files changed:** `ui/src/state/store.ts`, `ui/src/features/clue/index.tsx`, `ui/src/features/check/index.tsx`, `ui/tests/e2e/check.spec.ts`, `.ui-fix-memory.md`.
- **Fix:** The ordered-word and absolute-role board comparator is now shared by the clue and check flows. A check response is accepted only if the current live board exactly matches the request snapshot; otherwise the already-cleared result state remains empty, no response is cached, no checked clue or map hint is set, and a Hebrew retry toast explains that the board changed.
- **Verification:** A deterministic Playwright regression gates delivery of the resolved real check response, proves its request contains `אריה`, changes that tile to `chosen`, releases the response, and verifies the retry toast plus zero result, old ranked row, checked clue, and map hint. `PW_PORT=5155 npm run typecheck` passed and `PW_PORT=5155 npx playwright test tests/e2e/check.spec.ts tests/e2e/clue.spec.ts` passed all 17 focused tests (9 Check, 8 Clue). `git pull --no-rebase origin feature/ui` reported the branch up to date with no conflicts, then `PW_PORT=5155 npm run test:e2e` passed all 57 tests. A separate manual Playwright replay at 1320×900 on port 5155 observed the old request containing `אריה`, then the retry toast with zero result, null checked clue, and no map hint after the lifecycle change; a fresh resubmission excluded `אריה`, rendered the result, set checked clue `טבע`, and updated the semantic map. `git diff --check` passed; package and lock files were unchanged.

## UIFIX-009 — Direct word editing is overwritten by in-flight OCR

- **Severity:** High (manual input data loss)
- **Manual reproduction:** On port 5155, upload the 1px PNG through `photo-input-board`, wait for `מזהה את מילות הלוח…`, and directly fill `ocr-cell-0` with `בדיקה` without pressing the manual-skip button. Wait for the real OCR worker to finish.
- **Observed evidence:** The input initially contained `בדיקה`, but when recognition completed the UI reported success and replaced that manual value with the OCR result (empty for the 1px image).
- **Root cause:** The board OCR request-ID guard was invalidated by manual skip, newer uploads, and unmounting, but the editable word-grid handler did not invalidate an active attempt. Direct typing could therefore race with an older worker that still had permission to replace words, confidences, and OCR status or show an error toast.
- **Files changed:** `ui/src/features/photo/index.tsx`, `ui/tests/e2e/photo.spec.ts`, `.ui-fix-memory.md`.
- **Fix:** All word-input edits now use one helper. If OCR is actively recognizing, the first edit advances the attempt ID and immediately returns the status to the ready manual state before applying the word and its corrected confidence; the stale worker may finish but cannot modify words, confidences, status, or toasts. Normal recognition without edits and edits made outside the recognizing state retain their existing behavior.
- **Verification:** A deterministic Playwright regression reuses the delayed real OCR-worker helper, edits `ocr-cell-0` while recognition is active, confirms the ready state appears immediately, releases the Hebrew trained-data request, observes the actual worker recognition resolution, and verifies `בדיקה` remains with the ready status and no stale toast. `PW_PORT=5155 npm run typecheck` passed and `PW_PORT=5155 npx playwright test tests/e2e/photo.spec.ts` passed all 12 PhotoSetup tests. `git pull --no-rebase origin feature/ui` reported the branch up to date with no conflicts, then `PW_PORT=5155 npm run test:e2e` passed all 58 tests. A separate manual Playwright replay at 1320×900 on port 5155 used the real worker and observed `בדיקה` plus ready status immediately and after five seconds, with zero toast. `git diff --check` passed; package and lock files were unchanged.
